## High

### H1 — `About.tsx` is marked `"use client"` with no client-side code
**File:** `src/sections/About.tsx:1`
**Issue:** The `"use client"` directive forces this component and every module it imports (13 SVG icons, two images, `SectionHeader`, `Card`, `CardHeader`, `ToolboxItems`, `TechIcon`) into the client JavaScript bundle, even though the component contains zero hooks, event handlers, or browser APIs. On a static portfolio this forfeits React Server Component rendering benefits and adds unnecessary JS to the initial parse cost.
**Fix:** Remove the `"use client"` directive. The CSS animations (`animate-move-left`) run via Tailwind/CSS, not React state, and need no JS.

### H2 — Missing `priority` prop on the above-the-fold Hero image
**File:** `src/sections/Hero.tsx:67-71`
**Issue:** The memoji avatar is the first visible image users see, but no `priority` prop is set on the `<Image>` component. Next.js will lazy-load it by default, delaying Largest Contentful Paint.
**Fix:** Add `priority` to that `<Image>`. Also add `sizes="100px"` since the image is rendered at a fixed `size-[100px]` — without `sizes`, Next.js generates oversized srcset candidates.

### H3 — Missing `sizes` prop on all project card images
**File:** `src/sections/Projects.tsx:313-318`
**Issue:** Four project screenshots are rendered via `<Image>` with no `sizes` attribute. Next.js defaults to `100vw`, causing it to serve and decode a full-viewport-width image even though on desktop the image occupies roughly one column of a two-column grid (~600px). This pulls unnecessarily large files over the network.
**Fix:** Add `sizes="(max-width: 1200px) 100vw, 600px"` to each project image.

### H4 — Project card images have no defined dimensions (unoptimized path)
**File:** `src/sections/Projects.tsx:313-318`
**Issue:** The images use `StaticImageData` imports so Next.js knows the intrinsic size, but the element uses CSS classes like `lg:absolute lg:h-full lg:w-auto lg:max-w-none` with no explicit width/height props. Combined with missing `sizes`, Next.js cannot pick the right source candidate, defaulting to the largest one.
**Fix:** Add explicit `fill` prop with a positioned parent or explicit `width`/`height` props matching the rendered layout, alongside `sizes`.

### H5 — All `target="_blank"` links are missing `rel="noopener"`
**File:** `src/sections/Projects.tsx:202, 263` | `src/sections/Footer.tsx:36` | `src/sections/Hero.tsx:92, 99`
**Issue:** All external links use `rel="noreferrer"` only. While `noreferrer` implies `noopener` in modern browsers, `noopener` alone does not imply `noreferrer`. The explicit `noopener` is still recommended for defense in depth per MDN and is the pattern expected by ESLint security plugins. Currently fine in all major browsers, but the pattern is inconsistent with best practice.
**Fix:** Change all external link `rel` values to `rel="noopener noreferrer"`.

### H6 — Broken image imports in `Testimonials.tsx`
**File:** `src/sections/Testimonials.tsx:1-5`
**Issue:** The file imports five images (`memoji-avatar-1.png` through `memoji-avatar-5.png`) that do not exist in `src/assets/images/`. The build will fail if this file is ever imported. The component currently renders only `<div>Testimonials Section</div>` and is not imported anywhere, masking the build error.
**Fix:** Delete the file, or if testimonials are a planned feature, stub it without broken imports.

### H7 — `next.js` version behind on a recently-patched minor
**File:** `package.json:19`
**Issue:** Version is pinned to `^14.2.30`. As of early 2026, `14.2.35` is the latest stable 14.x patch, which includes fixes for CVE-2025-55184 (DoS via malformed App Router request) and related advisories. The project's stable 14.x is not affected by the App Router RSC RCE (CVE-2025-66478, which targets canary/15.x), and CVE-2025-29927 is already patched at 14.2.25. Staying current on the 14.x patch train is low risk and good hygiene.
**Fix:** Run `npm install next@14.2.35` (or the current latest 14.x) and test the build.

### M13 — Duplicate SVG gradient `id` rendered multiple times on the same page
**File:** `src/components/TechIcon.tsx:6`
**Issue:** Each `<TechIcon>` renders a hidden `<svg>` element containing `<linearGradient id="tech-icon-gradient">`. `ToolboxItems` renders 13 icons twice (26 total `<TechIcon>` instances in `About.tsx`), putting 26 identical `id="tech-icon-gradient"` elements in the DOM. Duplicate `id` attributes are invalid HTML and can cause unpredictable gradient resolution across browsers, though in practice all copies are identical here so the visual impact is zero.
**Fix:** Render the gradient `<defs>` once in a shared location (e.g., `layout.tsx` or a `<GlobalSvgDefs>` component) and reference it from individual icons.

### M14 — `scroll-behavior: smooth` applied globally without `prefers-reduced-motion` guard
**File:** `src/app/globals.css:20`

```css
@media (prefers-reduced-motion: no-preference) {

```
For the Tailwind animations, add `motion-reduce:animate-none` or `motion-reduce:pause` classes alongside each animation class, or configure the `tailwind.config.ts` keyframes to respect the media query.

### L1 — `metadata.description` is empty
**File:** `src/app/layout.tsx:15`
**Issue:** `description: ""` produces `<meta name="description" content="">` — an empty description is worse than no description for SEO, as search engines may penalize it or substitute their own poor-quality snippet.

### L2 — Missing `viewport` metadata export
**File:** `src/app/layout.tsx`
**Issue:** Next.js 14 App Router separates viewport metadata from the `metadata` object. No `viewport` export exists. Without it, Next.js uses its built-in default, which is fine for most cases, but explicitly exporting it is required to set `themeColor` and ensures `viewport` does not appear in the `metadata` export (which is deprecated behavior).
**Fix:**
```ts
export const viewport: Viewport = {
  width: 'device-width',
  initialScale: 1,
  themeColor: '#0a0a0a',
};
```

### L7 — `next.config.mjs` has no `Content-Security-Policy` or security headers
**File:** `next.config.mjs`
**Issue:** No HTTP security headers are configured: no `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, or `Referrer-Policy`. For a static marketing site hosted on Vercel, the risk is low, but CSP in particular defends against script injection via third-party dependencies and protects the EmailJS contact form.
**Fix:** Add a `headers()` function to `next.config.mjs`. At minimum:
```js
async headers() {
  return [{
    source: '/(.*)',
    headers: [
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'X-Frame-Options', value: 'SAMEORIGIN' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
    ],
  }];
},
```
CSP is more involved due to inline styles (Tailwind) and `@emailjs/browser`'s CDN calls, but even a permissive `default-src 'self'` report-only policy is a good starting point.

### L8 — `tailwind.config.ts` breakpoints override defaults without documenting intent
**File:** `tailwind.config.ts:11-15`
**Issue:** The `screens` block uses `theme.screens` (replacing all defaults) rather than `theme.extend.screens` (merging). This means the standard `xl` and `2xl` breakpoints don't exist. This is a deliberate design choice given the three-breakpoint system used, but it's easy to accidentally add a component with `xl:` or `2xl:` classes that silently does nothing.
**Fix:** Add a comment: `// Intentionally replaces defaults — use sm/md/lg only.` This is a low-risk, documentation-only change.

### L9 — `ToolboxItems.tsx` uses `[...new Array(2)].fill(0)` as a loop primitive
**File:** `src/components/ToolboxItems.tsx:16` | `src/sections/Tape.tsx:31`
**Issue:** The pattern `[...new Array(2)].fill(0).map((_, index) => ...)` is a common antipattern used to repeat JSX twice for infinite scroll loops. It works, but `Array.from({ length: 2 })` is cleaner and more readable. More importantly, the two copies are the only items that ever exist, meaning the "infinite" animation will visibly loop (restart) rather than appearing continuous unless the CSS `translateX(-50%)` animation is precisely calibrated. Worth validating visually at various viewport widths.
**Fix:** Minor cleanup: replace with `Array.from({ length: 2 })`. Verify the loop is visually seamless.

### L10 — `Header` component uses `<div>` as the outermost element instead of `<header>`
**File:** `src/sections/Header.tsx:3`
**Issue:** The outermost element is a `<div>` but it acts as the site header. `Footer.tsx` correctly uses `<footer>`. Using semantic `<header>` allows assistive technologies to identify and navigate to the landmark.

### L11 — `page.tsx` root element is a plain `<div>` with no landmark role

## Configuration Notes

### `tsconfig.json`
`strict: true` is set — good. `skipLibCheck: true` is acceptable for a project without custom declaration files. `allowJs: true` is included but there are no `.js` source files under `src/`; this is harmless but could be removed to tighten the config.

### `tailwind.config.ts`
The `fontFamily` values use unquoted template literals (`var(--font-sans)`) where the Tailwind docs show string values — this works because CSS custom properties resolve at runtime, but the font family stack fallback is absent. If the CSS variable fails to load, the browser falls back to the browser default rather than a system sans-serif stack.

Fix:
```ts
fontFamily: {
  sans: ['var(--font-sans)', 'system-ui', 'sans-serif'],
  serif: ['var(--font-serif)', 'Georgia', 'serif'],
},
```

### `next.config.mjs`
No `images` config is present (no `remotePatterns`, no `formats`). All images are local static imports, so this is correct — no remote image domains are needed. Adding `formats: ['image/avif', 'image/webp']` would enable AVIF delivery for browsers that support it, further reducing payload for the project screenshots.
